# clemini 협업 효율 연구 — v0 (260622 / claude)

> 방법: Claude가 틀을 잡고, Gemini(agy --print)에 1차 분석을 위임한 뒤 Claude가 종합. (clemini 자체를 dogfooding.)
> 실증은 별도 빌드 태스크를 벤치마크로 진행했고, 여기엔 **clemini(도구)에 대한 발견만** 남긴다. 벤치마크 앱 자체의 설계·내용은 이 레포에 두지 않는다.

## 0. 전제 변화 (Claude Max 5x)
Claude 토큰은 더는 병목이 아니다. "효율" = Claude 토큰 최소화 ❌ → **결합 품질 ÷ (시간 + 비용) 최대화** ⭕. 이긴다는 건 **지연·커버리지·Claude 컨텍스트 보존**.

## 1. 논제 (Thesis)
**clemini = Claude 앞단의 "압축·병렬·교차비판" 깔때기.**
Gemini는 **넓은 깔때기**(큰 컨텍스트로 대량 흡수, 병렬 실행, 1차 비판), Claude는 **날카로운 끝**(종합·판단·편집). agy가 읽기·추론 전용(편집 불가)이라 이 분담이 자연스럽다.

## 2. 핵심 협업 패턴 5
| 패턴 | 내용 | 최대 위험 |
|---|---|---|
| **A. 컨텍스트 오프로드** | 대용량(코드베이스·문서·로그)을 Gemini가 흡수→구조화 추출→Claude는 추출본만 작업. | 요약이 핵심 의존성·설계 의도 누락 → Claude 오판 |
| **B. 병렬 팬아웃** | 독립 질문 N개를 agy 동시 호출→수집→Claude 종합. 지연 단축. | 호출당 ~25k 오버헤드 ×N |
| **C. 다모델 교차비판** | Gemini 독립 1차 반론(다른 모델=다른 맹점)+Claude 최종. | 환각 기반 허위 버그(False Positive)로 불필요 재작업 |
| **D. 적용 전 1차 검증** | Claude 산출물을 적용 전 Gemini가 규칙·정합성 1차 대조. | 정적 검증만으론 런타임 결함 통과 → 오판 |
| **E. 명세 압축 규율** | 닫힌 명세로 1차 정답률↑ → 왕복(=오버헤드) 최소화. | 과도 압축으로 맥락 부족 → Gemini 헛다리 |

> 공통 안전핀: Gemini 결과는 **1차**. 최종 적부·치명적·보안 검증은 Claude·결정론적 도구.

## 3. 측정 지표
1. **Claude 컨텍스트 절감**: Gemini가 흡수해 Claude가 안 읽은 토큰량.
2. **재작업/왕복 횟수**: 잘못된 Gemini 입력으로 발생한 Claude 재수정 빈도(낮을수록 신뢰↑).
3. **Lead Time**: 착수→조사→편집→최종검증 wall-clock (병렬 vs 직렬).
4. **품질 델타**: 결합 산출물 vs Claude-단독 루브릭 대조.

## 4. Gemini 절대 비위임
아키텍처 최종 결정 / 비밀(.env·키) 직접 취급 / 실제 파일 쓰기·git 반영 / 비가역 인프라·DB 조작. (agy 읽기전용 + 우리 규칙과 일치.)

## 5. 실증 발견 (별도 빌드 태스크를 벤치마크로 — 260622)
지식 밀도 높은 도메인 리서치 + 실제 빌드 1건을 clemini로 수행하며 관찰한 **도구 발견**:

- **패턴 A 검증(긍정)**: 도메인 리서치를 Gemini에 위임 → 장문 구조화 보고서 생성, Claude는 raw 검색 대신 브리프만 소비 → 컨텍스트 절감 큼. 핵심 주장은 Claude가 검색 1회로 검증 → 재작업 0.
- **패턴 C 보너스**: agy `--print`가 자체 멀티에이전트 안티테제를 무료로 돌림 — 교차비판이 위임 한 번에 딸려옴.
- **패턴 D 필수 재확인**: Gemini가 **전제를 잘못 깔 수 있음**(예: 존재하는 에셋/경로를 "없다"고 가정). 실재·최신·실시간 가능성은 **Claude가 검증**해야 정확.
- **[개선점] delegate.sh 아티팩트 회수**: agy `--print`는 복잡 산출물을 `~/.gemini/antigravity-cli/brain/<uuid>/*.md`에 쓰고 stdout엔 포인터만 줌. delegate.sh가 그 경로를 파싱해 본문 회수(cat)하도록.
- **[휴리스틱] 위임 ROI는 작업 종류에 좌우**: **넓은 조사·교차비판 = 위임** / **좁은 API조회·시각 이터레이션·실재검증 = Claude 직접**. 호출당 ~25k 오버헤드라 좁은 작업 위임은 순손해.
- **[개발루프] 검증 도구**: GPU/시각 산출물 검증엔 헤드리스 프리뷰가 부적합 → 실제 환경 캡처가 정답. (clemini는 위임 도구일 뿐, 최종 시각 검증은 Claude가 실환경에서.)

**결론**: clemini 핵심 가치(리서치 오프로드 + 교차비판)는 실증됨. 단 ROI는 "무엇을 위임하느냐"에 강하게 의존 — 넓은 조사엔 강하고, 좁은 조회·이터레이션엔 오버헤드만.
